# /installmcp — установка MCP-серверов из дистрибутива

Команда выполняет первичную установку MCP-серверов для 1C, используя **локальный распакованный дистрибутив**, который пользователь скачал и распаковал сам. Все шаги (образы, тома, переменные окружения, лицензионные ключи, порядок запуска) описаны **внутри дистрибутива** — команда не выдумывает их и не подтягивает извне, а исполняет ровно то, что в нём написано.

Для проверки уже установленных серверов используй `/checkmcp`. Для обновления уже установленного набора — `/updatemcp`.

## Шаги

### 1. Запросить путь к дистрибутиву

Спросить у пользователя один параметр:

> Укажите путь к каталогу с распакованным дистрибутивом MCP-серверов 1C (например, `D:\dist\mcp-1c`).

Если пользователь не дал путь — остановиться и попросить его. Не угадывать, не искать «по умолчанию», не предлагать `git clone` — дистрибутив должен лежать локально и уже быть распакован.

### 2. Проверить каталог

После получения пути:

1. Убедиться, что каталог существует и доступен на чтение.
2. Перечислить его содержимое (PowerShell):

   ```powershell
   $dist = '<DIST_PATH>'
   if (-not (Test-Path -LiteralPath $dist)) { throw "Каталог не найден: $dist" }
   Get-ChildItem -LiteralPath $dist -Force | Select-Object Mode, Name, Length | Format-Table -AutoSize
   ```

3. Найти файл с инструкциями по установке. Кандидаты в порядке приоритета:
   - `INSTALL.md`, `install.md`
   - `README.md`, `readme.md`
   - `SETUP.md`, `setup.md`
   - `docs/INSTALL.md`, `docs/README.md`
   - любой `.md` / `.txt` верхнего уровня, в названии которого есть `install` / `setup` / `quickstart` / `getting-started`

   Если ничего не нашлось — остановиться и сообщить пользователю: «В каталоге `<DIST_PATH>` не найдено файла с инструкцией по установке (`INSTALL.md` / `README.md` / `SETUP.md`). Уточните путь или название файла.»

### 3. Прочитать инструкцию полностью

Прочитать найденный файл целиком (а не первые N строк). Если в инструкции есть ссылки на другие локальные файлы дистрибутива (`./scripts/...`, `./compose/docker-compose.yml`, `./env/...`, `./certs/...`, и т. п.) — открыть и эти файлы тоже, прежде чем что-либо запускать.

Источник правды на этом шаге — **только файлы внутри `<DIST_PATH>`**. Не подменять их инструкциями из `/checkmcp`, `https://docs.onerpa.ru/mcp-servery-1c`, `https://vibecoding1c.ru/mcp_server`, `content/mcp-servers.json` и т. п. Эти источники можно использовать только для свериться с уже выполненным шагом, не для его подмены.

### 4. Спланировать установку

До запуска команд:

1. Кратко (3–7 строк) пересказать пользователю, что именно будет сделано по инструкции дистрибутива: какие серверы поднимаются, какие тома/каталоги создаются, какие переменные окружения требуются, что нужно будет ввести руками (лицензионные ключи, токены, пути к выгрузке конфигурации и `bin` платформы).
2. Перечислить все недостающие данные одним списком и **запросить их у пользователя одним сообщением**, не дёргая по одному.
3. Если в инструкции упоминаются дополнительные требования среды (Docker Desktop, Neo4j, открытые порты, объём диска под индексы, доступ в интернет для `docker pull`) — явно перечислить их и убедиться, что они выполнены, до запуска.

### 5. Выполнить установку

Выполнить шаги ровно в том порядке, в каком они описаны в инструкции дистрибутива. Правила:

- Команды запускать из корня проекта, если в инструкции явно не сказано иначе.
- Если инструкция предлагает `docker compose up -d` / `docker run ...` / готовый `install.ps1` внутри дистрибутива — использовать именно его, а не собирать команду из таблицы `/checkmcp`.
- Не запускать ни `docker run`, ни `docker compose up` без подтверждения пользователя — образы могут весить несколько ГБ.
- Каждую тяжёлую команду показать пользователю до запуска (одна команда — одно сообщение или один блок).
- После каждого крупного шага кратко отчитаться (что запущено, какой контейнер/имя/порт), прежде чем переходить к следующему.

Если инструкция включает первичную RAG-индексацию (`1C-docs-mcp`, `1c-code-metadata-mcp`, `1c-graph-metadata-mcp`, `1c-ssl-mcp`) — предупредить пользователя, что первый запуск может занять десятки минут или часы, и показать команду для отслеживания (`docker logs -f <name>`).

### 6. Зарегистрировать серверы в активном инструменте

После того как контейнеры подняты:

1. Если в проекте есть `.ai-rules.json` — конфиг MCP активного инструмента (`.cursor/mcp.json` / `.mcp.json` / `.opencode/opencode.json` / `.codex/config.toml`) уже отрендерен установщиком 1c-rules; при необходимости перерендерить через `/update`.
2. Если установки 1c-rules не было — добавить серверы в MCP-конфиг активного инструмента вручную, по схеме из `adapters/<tool>.yaml → mcp.schema` или из инструкции дистрибутива (если в ней приведён готовый JSON/TOML-фрагмент — взять его).
3. Попросить пользователя перезапустить клиент (Cursor / Claude Code / Codex / OpenCode / Kilo Code), чтобы он переинициализировал MCP-сессию.

### 7. Финальная проверка

После перезапуска клиента запустить `/checkmcp`. Все серверы из дистрибутива должны получить статус **TOOLS_OK** (или хотя бы **HTTP_OK**, если индексация ещё идёт). Если что-то осталось в **TOOLS_MISSING** / **HTTP_DOWN** — вернуться к шагам 3–5 и сверить выполненное с инструкцией дистрибутива.

## Финальный отчёт

Короткая сводка пользователю:

- путь к использованному дистрибутиву;
- какой файл инструкции был прочитан;
- какие серверы реально подняты (имя контейнера, порт, образ);
- какие серверы пропущены и почему (например, нет `LICENSE_KEY`, нет выгрузки конфигурации, требуется отдельная установка Neo4j);
- следующие шаги, если индексация ещё идёт.

## Ограничения

- Команда **не** заменяет инструкцию дистрибутива и не дополняет её «своими» шагами. Если в инструкции чего-то нет — спросить пользователя, а не достраивать по памяти.
- Команда **не** скачивает дистрибутив сама. Предполагается, что пользователь скачал и распаковал его до вызова команды.
- Команда **не** запускает `docker run` / `docker compose up` без явного подтверждения пользователя.
- Графовый сервер (`1c-graph-metadata-mcp`) обычно требует отдельной установки Neo4j и индексации; если это шаг из инструкции — выполнять его строго по инструкции, а не по общим рецептам из `/checkmcp`.
